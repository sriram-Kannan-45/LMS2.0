1. SYSTEM OVERVIEW
================================================================================

The application is a Training Feedback Management System with three roles:
- ADMIN: Full system control (create trainers, trainings, manage participants)
- TRAINER: View assigned trainings, manage profile, reply to feedback, upload notes, create AI quizzes
- PARTICIPANT: Enroll in trainings, submit feedback, take quizzes

Tech Stack:
- Frontend: React + Vite + Tailwind CSS + Framer Motion + Chart.js + Recharts
- Backend: Express.js + Sequelize ORM + MySQL
- Auth: JWT (Bearer token)
- File Upload: Multer (profile images, notes)
- Notifications: In-app (Notification model) + Socket.IO

2.3 BACKEND ROUTES - trainerRoutes.js
--------------------------------------------------------------------------------

LOCATION: backend/src/routes/trainerRoutes.js
PREFIX: /api/trainer

ALL ROUTES REQUIRE: authenticateToken + roleMiddleware('TRAINER')

ENDPOINTS:

  GET /api/trainer/trainings
    → Returns trainer's assigned trainings with enrolled counts
    → Response: { trainings: [{ id, title, description, startDate, endDate, capacity, enrolledCount, availableSeats }] }

  GET /api/trainer/feedbacks
    → Returns all feedback for trainer's trainings
    → Includes: training title, participant name (or Anonymous), ratings, comments
    → Response: { feedbacks: [...], averageTrainerRating, averageSubjectRating }

  GET /api/trainer/profile
    → Returns trainer's user info + extended profile (TrainerProfile)
    → Response: { trainer: { id, name, email, username, phone, profile: { phone, dob, qualification, experience, imagePath } } }

  POST /api/trainer/profile
    → Create or update profile with image upload (multipart)
    → Accepts: phone, dob, qualification, experience, profileImage (file)
    → Upserts TrainerProfile record

  PUT /api/trainer/profile
    → Update profile with image upload (multipart)
    → Updates both User table (name, phone) and TrainerProfile table
    → Validates phone format, DOB format

  PUT /api/trainer/update
    → PRIMARY profile update endpoint (used by TrainerForm)
    → Multipart/form-data with profileImage
    → Validates: phone (7-15 digits), DOB (valid date)
    → Updates User table (name, phone) + TrainerProfile (phone, dob, qualification, experience, imagePath)
    → Response: { success, message, data: { user, profile } }

  GET /api/trainer/notifications
    → Returns trainer's notifications ordered by created_at DESC

  PUT /api/trainer/notifications/read
    → Marks all unread notifications as read

  PUT /api/trainer/change-password
    → Validates old password, requires new password >= 6 chars
    → Hashes new password with bcrypt (salt 10)

3.3 BACKEND CONTROLLER - adminController.js
--------------------------------------------------------------------------------

LOCATION: backend/src/controllers/adminController.js

FUNCTIONS:

  updateTraining(req, res)
    → Finds training by ID
    → Validates trainerId if provided (must be TRAINER role)
    → Updates: title, description, trainerId, startDate, endDate, capacity
    → Returns updated training with trainer name

  deleteTraining(req, res)
    → Finds training by ID
    → CASCADE DELETE: Feedback → Enrollment → Training
    → Response: { message: 'Training deleted successfully' }

  updateTrainer(req, res)
    → Finds user by ID with role=TRAINER
    → Validates email uniqueness if changed
    → Updates: name, email
    → Response: { message, trainer: { id, name, email, username } }

  deleteTrainer(req, res)
    → Finds user by ID with role=TRAINER
    → Unlinks trainings: sets trainerId=null for all trainer's trainings
    → Deletes user record
    → Response: { message: 'Trainer deleted successfully' }

  getStats(req, res)
    → Comprehensive statistics:
      - totalTrainings, completedTrainings, activeTrainings
      - totalTrainers, totalParticipants, pendingParticipants
      - totalEnrollments, totalFeedbacks, pendingNotes
      - avgTrainerRating, avgSubjectRating, satisfactionScore
      - ratingDistribution (1-5 breakdown)
      - enrollmentRate (enrollments/participants %)
    → Returns both flat properties AND nested data{} for compatibility

  getParticipants(req, res)
    → Supports: search (name/email/phone), status filter, pagination (limit/offset)
    → Excludes password from response
    → Response: { success, participants: [...], total, hasMore }

  sendReminders(req, res)
    → Finds training by ID
    → Gets enrolled participants
    → Filters out those who already submitted feedback
    → Creates bulk notifications for remaining participants
    → Response: { message: 'Sent X reminders.' }

  deleteParticipant(req, res)
    → Finds user by ID with role=PARTICIPANT
    → CASCADE DELETE: Enrollment → Feedback → User
    → Response: { message: 'Participant removed successfully' }

  exportFeedbacksCSV(req, res)
    → Fetches all feedbacks with training + trainer + participant info
    → Generates CSV with columns: ID, Training, Trainer, Participant, Trainer Rating, Subject Rating, Comments, Anonymous, Date
    → Sets Content-Type: text/csv, Content-Disposition: attachment
    → Handles anonymous feedback

  getTrainingStats(req, res)
    → Per-training statistics:
      - enrolledCount, feedbackCount
      - avgTrainerRating, avgSubjectRating
      - status (Upcoming/Ongoing/Completed based on dates)
    → Response: { trainings: [...] }

  getPendingParticipants(req, res)
    → Lists all PARTICIPANT users with status=PENDING
    → Response: { participants: [...], total }

  approveParticipant(req, res)
    → Finds PARTICIPANT with status=PENDING
    → Updates status to 'APPROVED'
    → Logs activity via ActivityService (USER_APPROVED)
    → Creates notification for participant: "Your account has been approved"
    → Response: { message, participant: { id, name, email, status } }

  rejectParticipant(req, res)
    → Finds PARTICIPANT with status=PENDING
    → CASCADE DELETE: Enrollment → Feedback → User (completely removes)
    → Response: { message: 'Participant rejected and removed successfully' }

6. DATABASE MODELS INVOLVED
================================================================================

User
  - id, name, email, username, password, phone, role (ADMIN/TRAINER/PARTICIPANT), status (PENDING/APPROVED)
  - Associations: hasOne TrainerProfile, hasMany Training (as trainer), hasMany Enrollment, hasMany Feedback, hasMany Notification

TrainerProfile
  - id, userId (FK → User), phone, dob, qualification, experience, imagePath
  - Stores extended trainer info beyond User table

Training
  - id, title, description, trainerId (FK → User), startDate, endDate, capacity, createdBy
  - Associations: belongsTo User (as trainer), hasMany Enrollment, hasMany Feedback

Enrollment
  - id, participantId (FK → User), trainingId (FK → Training), status (ENROLLED)
  - Links participants to trainings

Feedback
  - id, trainingId (FK → Training), participantId (FK → User), trainerRating, subjectRating, comments, anonymous, trainerResponse, submitted_at
  - Participant feedback with optional trainer reply

Notification
  - id, userId (FK → User), message, type, isRead, created_at
  - In-app notifications for all roles

Note
  - id, title, content, trainerId, trainingId, status (PENDING/APPROVED/REJECTED), filePath, created_at
  - Trainer-uploaded resources requiring admin approval

ActivityLog
  - id, userId, userName, action, entityType, entityId, details, created_at
  - Audit trail for admin actions

SurveyQuestion
  - id, trainingId (optional), questionText, questionType (TEXT/RATING/MULTIPLE_CHOICE), options[]
  - Dynamic feedback survey questions

7. AUTHENTICATION & AUTHORIZATION
================================================================================

MIDDLEWARE:
  - authenticateToken: Verifies JWT Bearer token, attaches req.user { id, role, name, email }
  - roleMiddleware(role): Checks req.user.role matches required role
  - optionalAuth: Attaches user if token present, continues without error if not

TOKEN FORMAT:
  Authorization: Bearer <jwt_token>

ROLE HIERARCHY:
  ADMIN  → Full access to all admin routes + can access training routes
  TRAINER → Access to /api/trainer/* routes only
  PARTICIPANT → Access to enrollment, feedback, quiz routes

PARTICIPANT REGISTRATION FLOW:
  1. User registers → status='PENDING'
  2. Admin sees in Pending Approval tab
  3. Admin approves → status='APPROVED' → user can now login
  4. Admin rejects → user record deleted entirely

PASSWORD SECURITY:
  - Hashed with bcrypt (salt rounds: 10)
  - Trainer can change own password via PUT /api/trainer/change-password
  - Requires old password verification
