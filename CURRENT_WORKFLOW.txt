This document describes the workflow as it actually exists in the codebase
(routes, controllers, and models that are wired and running today).


--------------------------------------------------------------------------------
1. ARCHITECTURE
--------------------------------------------------------------------------------

  Browser (React + Vite, port 5173)
        |
        |  REST (fetch) + Socket.IO (real-time)
        v
  Node / Express backend (port 3001)
        |
        |  HTTP (server-to-server only)
        v
  Python AI service (FastAPI, port 8000)   <-- quiz generation from documents

  Datastore : MySQL via Sequelize ORM
  Auth      : JWT (Bearer token), role claim in token
  Realtime  : Socket.IO (notifications, proctoring, live sessions)
  Files     : local /uploads (ai-docs, avatars, notes, trainer) + Cloudinary

  Rule: the frontend NEVER calls the Python service directly. Node proxies it.

  Start everything: start-all.bat / start-all.ps1
  Backend dev:      cd backend && npm run dev   (nodemon, auto-creates tables)


--------------------------------------------------------------------------------
2. ROLES
--------------------------------------------------------------------------------

  ADMIN        Creates trainers, trainings; approves participants; analytics.
  TRAINER      Authors lessons/quizzes/assessments; reviews & publishes results.
  PARTICIPANT  Enrolls, learns, takes quizzes, submits assessments, views results.

  Enforced by:
    middleware/auth.js   -> verifies JWT, sets req.user
    middleware/roles.js  -> roleMiddleware('ADMIN', 'TRAINER', ...)

3. AUTH & ONBOARDING WORKFLOW
--------------------------------------------------------------------------------

  Register/Login          POST /api/auth/register, /api/auth/login
  Change password         POST /api/auth/change-password           (auth)
  Forgot password (OTP)   POST /api/auth/forgot-password/send-otp
                          POST /api/auth/forgot-password/verify-otp
                          POST /api/auth/forgot-password/reset
                          (OTP emailed via nodemailer; expired rows cleaned
                           by a background job every 5 min)

  Participant approval flow:
    1. Participant registers          -> user.status = PENDING
    2. Admin reviews pending list      GET  /api/admin/pending-participants
    3. Admin approves / rejects        POST /api/admin/approve-participant/:id
                                       POST /api/admin/reject-participant/:id
    4. On approval -> status APPROVED, participant can enroll.

  A default admin (admin@test.com / admin123) is auto-created on first boot.

6. AI QUIZ WORKFLOW  (existing, document-driven)   /api/ai-quiz
--------------------------------------------------------------------------------

  TRAINER
    1. Upload a document (PDF/DOCX/TXT)   POST /trainer/upload-document
         -> text extracted (pdf-parse / mammoth) -> sent to Python AI service
         -> AIQuiz + AIQuestion rows generated.
    2. List / edit quizzes                GET  /trainer/quizzes
                                          PUT  /trainer/quiz/:id
  PARTICIPANT
    3. List available quizzes             GET  /participant/quizzes
    4. Start an attempt                   POST /participant/start/:quizId
    5. Submit answers                     POST /participant/submit/:attemptId
         -> auto-scored -> QuizAttempt/QuizAnswer/QuizResult rows.
    6. Leaderboard                        GET  /leaderboard/:quizId

  Models: AIDocument, AIQuiz, AIQuestion, QuizAttempt, QuizAnswer, QuizResult.


--------------------------------------------------------------------------------
7. PROCTORING WORKFLOW   (secure exams)   /api/proctor
--------------------------------------------------------------------------------

  Session lifecycle (participant):
    start -> activate -> heartbeat (kept alive) -> answers autosave
          -> finalize/submit -> terminate
  Locked endpoints require X-Proctor-Session-Token (sessionLock middleware).
  Violations & activities are recorded; trainer can monitor a quiz live
  (GET /quiz/:quizId/monitor) and force-terminate a session.
  Background reaper expires stale sessions every 60s.

  Models: ExamSession, Violation, DeviceFingerprint, ProctorActivity,
          AssessmentSession (session-lock).


--------------------------------------------------------------------------------
8. LESSON WORKFLOW  (NEW — lesson / quiz / assessment gating)   /api/lessons
--------------------------------------------------------------------------------

  A lesson belongs to a training. "Assigned participants" = participants
  enrolled (status ENROLLED) in that training.

  TRAINER (role TRAINER or ADMIN)
    Create lesson                 POST /api/lessons
    List own lessons              GET  /api/lessons/trainer
    Attach quiz to lesson         POST /api/lessons/:lessonId/quizzes
                                       { quizId, isMandatory }   (reuses AIQuiz)
    Create assessment             POST /api/lessons/:lessonId/assessments
    Dashboard (per quiz)          GET  /api/lessons/:lessonId/dashboard
         -> { enrolled, completed, pending, canPublish } for each quiz
    Publish quiz results          POST /api/lessons/quizzes/:lessonQuizId/publish
         -> GATE: 409 "Results cannot be published until all assigned
            participants complete the quiz." while pending > 0.
         -> on success: result_status = PUBLISHED, all participants notified.
    Review assessment subs        GET  /api/lessons/assessments/:assessmentId/submissions
    Grade a submission            PUT  /api/lessons/submissions/:submissionId/grade
                                       { score, feedback }  -> status REVIEWED
    Publish assessment result     POST /api/lessons/submissions/:submissionId/publish
                                       -> status PUBLISHED, participant notified

  PARTICIPANT (role PARTICIPANT)
    List lessons                  GET  /api/lessons/participant
    Mark content viewed           POST /api/lessons/:lessonId/view
    Complete a quiz               POST /api/lessons/quizzes/:lessonQuizId/complete
         -> stores score (HIDDEN), responds ONLY:
            "Quiz completed successfully. Your responses have been submitted."
    Submit an assessment          POST /api/lessons/assessments/:assessmentId/submit
                                       { content, fileUrl }
    Quiz result status            GET  /api/lessons/quizzes/:lessonQuizId/result
         -> HIDDEN: "Your quiz has been submitted. Results will be available
            once published by the trainer."
         -> PUBLISHED: returns score.
    Assessment result status      GET  /api/lessons/assessments/:assessmentId/result
         -> hidden message until trainer publishes, then score + feedback.

  RESULT VISIBILITY
    Quiz scores and assessment grades are stored on submission but stay HIDDEN
    from participants until the trainer explicitly publishes them. A trainer
    can only publish quiz results once EVERY assigned participant has completed.

  LESSON COMPLETION LOGIC
    A lesson becomes COMPLETED for a participant only when:
      (a) lesson content has been viewed, AND
      (b) all MANDATORY quizzes are COMPLETED, AND
      (c) all MANDATORY assessments are SUBMITTED (REVIEWED/PUBLISHED also count).
    Recomputed automatically after view / quiz-complete / assessment-submit.

  STATUS FIELDS
    Quiz   (quiz_progress.status)         NOT_STARTED | IN_PROGRESS | COMPLETED
    Assess (assessment_submissions.status) NOT_STARTED | SUBMITTED | REVIEWED | PUBLISHED
    Lesson (lesson_progress.status)        NOT_STARTED | IN_PROGRESS | COMPLETED
    Result (lesson_quizzes.result_status)  HIDDEN | PUBLISHED

  Models: Lesson, LessonQuiz, LessonAssessment, AssessmentSubmission,
          QuizProgress, LessonProgress.

9. NOTIFICATIONS & REAL-TIME
--------------------------------------------------------------------------------

  Service: services/notificationService.js (DB row + Socket.IO emit).
  Channels: user_<id> room, role_<ROLE> room.
  Endpoints: GET /api/notifications, /unread/count
             PUT /api/notifications/:id/read, /read-all
             DELETE /api/notifications/:id

  Lesson-workflow triggers:
    - All assigned participants finish a quiz  -> trainer is notified.
    - Trainer publishes quiz results           -> all participants notified.
    - Trainer publishes an assessment result   -> that participant notified.

11. END-TO-END EXAMPLE (lesson workflow)
--------------------------------------------------------------------------------

  Admin     creates trainer + training.
  Trainer   creates a lesson, attaches an AI quiz (mandatory), adds an
            assessment, all under that training.
  Participant enrolls in the training (now "assigned" to the lesson),
            reads content (view), takes the quiz (gets only the success
            message), submits the assessment.
  System    when the LAST assigned participant completes the quiz, the trainer
            is notified; lesson auto-marks COMPLETED once content+quiz+
            assessment conditions are met.
  Trainer   opens dashboard (sees enrolled/completed/pending=0), publishes
            quiz results; grades + publishes the assessment.
  Participant receives notifications and can now see scores/feedback.
